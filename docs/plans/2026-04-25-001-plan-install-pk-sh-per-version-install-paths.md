# Plan: install-pk.sh per-version install paths

## Context

`.claude/install-pk.sh` currently writes `pk` to `$HOME/.local/bin/pk`. On Debian/Ubuntu (the cloud sandbox base), `~/.profile` auto-prepends `$HOME/.local/bin` to PATH, so any `pk` left there by a previous session is on PATH for every future session — even sessions whose `install-pk.sh` pins a different `PK_VERSION`. The script's only install gate is `command -v pk >/dev/null 2>&1 && exit 0`, so the stale binary trips the gate and the pinned version is silently ignored.

Reproduced today: a sandbox running `install-pk.sh` pinned to `v0.14.1` had `/root/.local/bin/pk` at `v0.14.0` from a prior session; the gate exited at line 21 and `pk --version` kept reporting `0.14.0`.

The proposed fix: stop sharing one install path across versions. Give each `PK_VERSION` its own install directory and have each session prepend that directory to PATH via `CLAUDE_ENV_FILE`. Different sessions / projects each use their own version; there's no shared mutable location to fight over, and no version comparison needed.

`install-pk.sh` is shipped to downstream projects via `pk setup`, so the change goes in the embedded template at `internal/setup/template/install-pk.sh`. Each consuming project picks it up on its next `pk setup`.

## Approach

### Where each version goes

`$HOME/.local/share/pk/<PK_VERSION>/pk` — e.g., `$HOME/.local/share/pk/v0.14.1/pk`.

Chosen because it's outside `$HOME/.local/bin` (the auto-PATHed directory whose stickiness caused the bug), it's a standard XDG-data location for per-user app data, and binaries cache across sessions so re-using the same pinned version is offline.

### Install gate

Three branches, no version parsing:

1. **`pk` already on PATH → exit 0.** Honors the developer-machine workflow (e.g., `~/go/bin/pk` from `go install`). In sandboxes — the only place the body of this script actually runs — the new install path isn't auto-PATHed, so this branch only fires when the developer has put `pk` on PATH themselves.
2. **Version-specific binary already cached at `$install_dir/pk` → PATH-inject and exit 0.** Cheap; no network.
3. **Otherwise → download the pinned version to `$install_dir/pk`, verify checksum, install, PATH-inject.**

### Template patch

`internal/setup/template/install-pk.sh`. Replace lines 21–64 (early-exit through PATH-inject) with the block below. The header comment (lines 1–18) needs the install path updated; the trailing `git fetch --tags` block (lines 66–72) is unchanged.

```bash
command -v pk >/dev/null 2>&1 && exit 0

PK_VERSION="{{VERSION}}"
install_dir="$HOME/.local/share/pk/$PK_VERSION"
binary="$install_dir/pk"

if [ -x "$binary" ]; then
  [ -n "${CLAUDE_ENV_FILE:-}" ] && echo "export PATH=\"$install_dir:\$PATH\"" >> "$CLAUDE_ENV_FILE"
  exit 0
fi

if ! command -v sha256sum >/dev/null 2>&1; then
  echo "pk install: sha256sum not found — cannot verify binary integrity" >&2
  exit 1
fi

mkdir -p "$install_dir"

arch="$(uname -m)"
case "$arch" in
  x86_64) arch=amd64 ;;
  aarch64|arm64) arch=arm64 ;;
  *) echo "unsupported arch: $arch" >&2; exit 0 ;;
esac

base="https://github.com/markwharton/plankit/releases/download/${PK_VERSION}"
tmp="$(mktemp -t pk.XXXXXX)"
sums="$(mktemp -t pk-sums.XXXXXX)"
trap 'rm -f "$tmp" "$sums"' EXIT

curl -fsSL "$base/pk-linux-${arch}" -o "$tmp"
curl -fsSL "$base/checksums.txt" -o "$sums"

expected="$(awk -v n="pk-linux-${arch}" '$2 == n {print $1}' "$sums")"
if [ -z "$expected" ]; then
  echo "pk install: checksum not published for pk-linux-${arch}" >&2
  exit 1
fi

actual="$(sha256sum "$tmp" | awk '{print $1}')"
if [ "$expected" != "$actual" ]; then
  echo "pk install: checksum mismatch for pk-linux-${arch}" >&2
  echo "  expected: $expected" >&2
  echo "  actual:   $actual" >&2
  exit 1
fi

chmod +x "$tmp"
mv "$tmp" "$binary"

[ -n "${CLAUDE_ENV_FILE:-}" ] && echo "export PATH=\"$install_dir:\$PATH\"" >> "$CLAUDE_ENV_FILE"
```

Header comment edit: change "downloads the matching pk release into `$HOME/.local/bin`" to "downloads the matching pk release into `$HOME/.local/share/pk/<version>` and prepends it to PATH for the session."

## Operational note (existing sandbox)

The user's current sandbox has `$HOME/.local/bin/pk` (v0.14.1) from prior installs. After this change ships, the new template still exits at `command -v pk && exit 0` because that legacy binary is on PATH via the auto-PATHed `$HOME/.local/bin`. To activate the new per-version flow in that sandbox, run once:

```bash
rm $HOME/.local/bin/pk
```

The next session will then download to `$HOME/.local/share/pk/<PK_VERSION>/pk` and PATH-inject. Subsequent sessions short-circuit on the cached binary.

## Design dependencies (verified)

- **`CLAUDE_ENV_FILE` PATH injection takes effect for the current session** — confirmed against Claude Code's hooks documentation. Subsequent Bash tool calls and hook invocations in the same session see the prepended directory.
- **`$HOME/.local/bin` is auto-PATHed in cloud sandboxes** (via `~/.local/bin/env` sourced from `.profile`/`.bashrc`/`.zshrc`); `$HOME/.local/share` is not. Confirmed by inspection of the user's actual sandbox — this is exactly what motivates the version-isolated path choice.
- **Required tools present in cloud sandboxes**: `sha256sum`, `curl`, `awk` (mawk, not gawk — current and proposed scripts use only POSIX-portable awk patterns), `mktemp`. Confirmed by inspection.
- **`$HOME` persists across sessions; `CLAUDE_ENV_FILE` does not** — version-pinned binaries cache across sessions; PATH injection is per-session, eliminating cross-session leakage.
